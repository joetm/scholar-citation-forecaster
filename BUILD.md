# Reproducing the Citation Forecaster bundle

This document gives an AMO reviewer (or anyone else) the exact steps to
rebuild `dist/content.bundle.js` from source. The build is deterministic
given pinned dependencies — the output should byte-match the bundle
inside the submitted `.xpi`.

## Build environment

- **OS:** any platform that supports Node.js 18+ (tested on Ubuntu 22.04 / Linux x86_64)
- **Node.js:** v18.17.0 or newer (developed against v22.21.1)
- **npm:** v9 or newer (developed against v10.9.4)
- **Disk:** ~250 MB for `node_modules/` after install
- **Network:** required for the initial `npm ci` (no network used at runtime by the extension itself)

## Required tools (installed via `npm ci` from `package-lock.json`)

The build pulls these versions, all pinned exactly:

| Tool    | Version | Purpose                                                |
| ------- | ------- | ------------------------------------------------------ |
| esbuild | 0.21.5  | Bundles + minifies `src/content.js` and its imports    |
| eslint  | 8.57.1  | Lint (not strictly required to reproduce the bundle)   |
| jsdom   | 24.1.3  | Test-time DOM (not used at build time)                 |
| vitest  | 1.6.1   | Test runner (not used at build time)                   |
| web-ext | 7.12.0  | Packages `dist/` into a `.zip` via `npm run package`   |

No runtime dependencies — the extension itself is dependency-free.

## Build steps

From the project root:

```sh
# 1. Install pinned dependencies from package-lock.json
npm ci

# 2. Build the extension
npm run build
```

After step 2:

- `dist/content.bundle.js` — the minified, bundled content script
- `dist/manifest.json` — copied verbatim from `src/manifest.json`

The extension on disk is just those two files. The `dist/` directory contents are exactly what's packaged into the `.xpi` you reviewed.

## What `npm run build` does

It runs `node build/esbuild.config.js`, which:

1. Creates `dist/` if missing
2. Calls `esbuild.build()` with these options on `src/content.js`:
   - `bundle: true` — combines `src/content.js` and its `import`-ed siblings (in `src/dom/`, `src/stats/`) into a single output file
   - `format: 'iife'` — wraps the output in an immediately-invoked function expression so it runs as a content script
   - `target: 'firefox115'` — output syntax compatible with Firefox 115+
   - `minify: true` — minifies whitespace, identifiers, and dead code
3. Copies `src/manifest.json` to `dist/manifest.json`
4. (Convenience) also copies `dist/*` into `~/scholar-citation-projection/`. This is only useful for local development on snap-confined Firefox installs and has no effect on the AMO build artefact.

## Verifying the build matches the submitted bundle

After running the steps above, compare:

```sh
sha256sum dist/content.bundle.js
```

The hash should match the `content.bundle.js` inside the submitted `.xpi` (open the `.xpi` as a zip to inspect). esbuild 0.21.5 produces deterministic output given the same source and config.

## Optional checks

```sh
npm test     # 95 unit + integration tests
npm run lint # eslint over src/, test/, build/
```

Both should be clean.

## Repackaging from scratch

To reproduce the `.xpi` itself (not just the bundle):

```sh
npm run package
# produces web-ext-artifacts/citation_forecast-<version>.zip
```

This invokes `web-ext build --source-dir=dist --artifacts-dir=web-ext-artifacts --overwrite-dest` against the freshly built `dist/`.

## Files that are NOT machine-generated

All source files under `src/`, `test/`, and `build/` are handwritten. So is `package.json`. `package-lock.json` is generated by npm during `npm install` / `npm ci` and pins the dep tree for reproducibility.
